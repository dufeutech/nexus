# Design — slo-sloth-source-and-floor

## Context

Two services expose multi-window burn-rate SLOs generated by Sloth v0.16.0
(`prometheus/v1`). The full toolchain **already exists in the repo**:

- Sources: `monitoring/slo/tenant-router.slo.yaml`, `monitoring/slo/identity-sidecar.slo.yaml`
- Generator: `monitoring/slo/generate.sh` — runs the pinned `ghcr.io/slok/sloth:v0.16.0`
  image, writes `monitoring/prometheus/rules/*.rules.yaml`, then **stages** each into its
  chart at `deploy/helm/<chart>/files/slo/*.rules.yaml` (Helm embeds via `.Files.Get`).
- Validation: `monitoring/slo/check.sh` — promtool `check rules` (PromQL portability guard)
  + `test rules` (burn-rate unit tests in `monitoring/slo/tests/`).
- CI: `ci.yml` `monitoring-delivery` job runs `generate.sh`, `git diff --exit-code`
  (drift), and `check.sh`, then renders every chart under each `monitoring.delivery` mode.

The one thing missing is the **minimum-sample floor**. The SLI ratios carry none, so at
idle traffic the `severity: page` alerts fire on statistical noise — infra finding **N15**,
which paged on go-live day at ~0.035 req/s. (N15's claim that "the source spec is not in
the repo" was mistaken — only the floor was never authored into the existing sources.)

The prior change (`slo-low-traffic-guards`) floored the **hand-authored** threshold alerts
in `_monitoring.tpl` and synced the `service-slo-policy` requirement that mandates the
floor, but explicitly left the **generated burn-rate family** as the remaining gap. This
change closes it at the existing source.

Constraints:
- The floor must live in the *source* (`monitoring/slo/*.slo.yaml`), never the generated
  output (`# DO NOT EDIT`; reverted on the next `generate.sh`).
- `generate.sh`/`check.sh` are POSIX `sh`, docker-based, already handle Windows Git-Bash
  path quirks (`MSYS_NO_PATHCONV`) and the CI nonroot-bind-mount write
  ([[ci-windows-script-exec-and-bindmount]]).
- Edge-platform has no `files/slo/` → not a burn-rate consumer → out of scope.

## Goals / Non-Goals

**Goals**
- Idle-window pages on the four burn-rate alerts stop; objectives/budgets unchanged.
- Availability outage coverage preserved for the near-zero-traffic case.
- The burn-rate unit tests continue to pass and gain a below-floor suppression case.

**Non-Goals**
- No new source layout, generator, or CI — all already exist; do not create a rival.
- No change to objectives (99% latency / 99.9% availability), windows, or burn factors.
- No new telemetry, no producer redeploy.
- Not fixing the total-process-death gap (readiness metric goes *absent*, not `0`) — a
  pre-existing platform/infra concern (a target-absent / `up == 0` alert), noted as
  follow-up.

## Decisions

### D1 — Author the floor into the existing Sloth source; keep Sloth (Adopt).

Burn-rate math (~10–13 interlocking rules per SLO) is correctness-critical → Adopt over
Build, and Sloth is already the adopted tool with a working generate/validate/drift
pipeline. The change is purely additive to the two existing `*.slo.yaml` sources; the
existing `generate.sh` propagates it to both output locations and both delivery forms.

### D2 — Floor authored in the SLI `total_query`, per-window, as a minimum sample **count** (industry-standard model); default 60 samples.

Appended to each SLO's `total_query` so the whole ratio is gated without touching the
error term or the events form:
```
total_query: (<existing total>) and (sum by (deployment_environment_name) (increase(<denom>[{{.window}}])) > 60)
```
`sloth generate` substitutes `{{.window}}` per window, so every `slo:sli_error:ratio_rateXX`
recording rule is floored on its own window's sample count. Below 60 samples the
denominator is empty → the SLI yields no series → the `(short) and (long)` alert cannot
fire. No alert edit needed.

- **Floor value model — minimum sample count, `increase(<denom>[{{.window}}]) > 60`, one
  literal per objective.** Industry standard (GitLab metrics-catalog
  `minimumSamplesForMonitoring`), and it *generalizes the repo's existing threshold* rather
  than adding a rival: `routingMinRps/enrichMinRps: 0.2` is documented as "≈ 60 req / 5m",
  so `min_samples = 60` reproduces the 0.2-rps floor on the 5m window exactly and scales to
  lower rps floors on longer windows (6h → ~0.0028 rps, 3d → ~0.0002 rps).
- **Why not uniform per-second (0.2 rps on every window):** a flat rate over-floors long
  windows and would suppress the long-window ticket arm for a low-but-steady tenant even
  though it has ample samples over 6h/3d. Sample-count keeps low-traffic objectives
  monitored on long windows while silencing short-window pages.
- **`min_samples = 60` is tunable per objective**; raise toward GitLab's 3600 if 60 proves
  thin for nexus traffic. Default 60 keeps the operator-validated anchor.
- **Denominator = the objective's own total series** (already present in `total_query`):
  `router_ext_proc_requests_total` (routing availability),
  `router_ext_proc_duration_seconds_count{result="hit"}` (routing latency), and the
  `sidecar_ext_proc_*` equivalents with the identity result-matchers. No new metric.

### D3 — Floor both latency and availability; availability is backstopped by the readiness gauge.

Both SLOs show the same idle-noise class (cold-start `not_ready` blips spike the
availability ratio just as first-hit slowness spikes latency). Flooring availability is
safe because a **traffic-independent** backstop already exists: `NexusRoutingNotReady` /
`NexusIdentitySidecarNotReady` (`router_ready == 0`, `severity: critical`, `for: 10m`) fire
on sustained unreadiness regardless of request volume (satisfies the new
`service-slo-policy` backstop requirement).

- **Rejected — latency-only:** leaves availability idle-pages unfixed for no safety gain,
  since the backstop already covers the volume-independent outage.
- **Residual gap (out of scope):** if the process dies entirely, `router_ready` goes
  *absent* (not `0`), so neither the floored availability SLO nor `== 0` fires. Pre-existing;
  belongs to a platform-layer target-absent / `up == 0` alert. Flag in N15 as follow-up.

### D4 — Reuse the existing source/generation/CI; extend the burn-rate unit tests.

The floor goes into `monitoring/slo/*.slo.yaml`; `monitoring/slo/generate.sh` regenerates
`monitoring/prometheus/rules/` and re-stages the Helm copies. `check.sh` (promtool) and the
CI `monitoring-delivery` drift/validation job already exist — nothing added there.

- **Test impact (must fix):** the floor changes what counts as "enough traffic to have an
  SLI." The existing `tests/tenant-router.slo_test.yaml` test 1 drives `not_ready` at
  10/min → `increase(total[5m]) = 50` < 60 → below the floor → the SLI is suppressed and its
  asserted page/ticket vanish. Fix: raise test 1's synthetic traffic above the floor (still
  a total-outage ratio of 1.0), keeping staging above the floor too. Then **add a
  below-floor suppression case**: near-idle traffic with a bad ratio asserts **no alert
  fires** — the direct N15 behaviour, now covered by a test.

## Decisions (build-vs-adopt, /opsx:decide)

### Decision: Multi-window burn-rate rule generation — Adopt Sloth v0.16.0 (already adopted)

- **Status**: approved
- **Why**: Correctness-critical burn-rate math; Sloth is already the adopted generator with
  a working generate/validate/drift pipeline pinned to v0.16.0. The change is additive to
  its existing sources — no reason to migrate or hand-author.
- **Considered**: Migrate to Pyrra (rewrites working output/wiring for no requirement);
  hand-author in `_monitoring.tpl` (Build — high-surface, drift-prone).
- **Isolation**: floor lives in `monitoring/slo/*.slo.yaml`; Sloth invoked only by the
  existing `generate.sh`; output vendored behind `.Files.Get`.

### Decision: Min-sample floor model — Extend the repo's 60-sample threshold via the GitLab sample-count method

- **Status**: approved
- **Why**: Industry-standard per-window minimum sample **count**
  (`increase(<denom>[{{.window}}]) > 60`); `min_samples = 60` **is** the repo's existing
  significance bar (`routingMinRps: 0.2` ≈ "60 req / 5m"), reproducing today's 5m floor
  exactly while keeping low-traffic objectives monitored on long windows.
- **Considered**: Uniform 0.2 rps per window (over-floors long windows); GitLab's default
  3600 samples (too high for nexus scale — 60 kept tunable toward it).
- **Isolation**: one `60` literal per objective in the source `total_query`; expanded to all
  7 windows by generation. Single source of truth for the burn-rate floor.

## Risks / Trade-offs

- **Floor suppresses the existing test-1 assertions** (D4) → raise that test's traffic above
  the floor and add a below-floor suppression case; `check.sh` must stay green.
- **`min_samples = 60` may be too thin at nexus scale** (D2) → tunable per objective; raise
  toward 3600 if noisy. Default 60 preserves the operator-validated 0.2-rps-over-5m anchor.
- **Availability floor + absent readiness metric** (D3) → residual, out of scope; recorded so
  it isn't mistaken for covered.
- **Generated-output diff churn** → the only intended change is the floor `and (... > 60)` on
  each `slo:sli_error:ratio_rateXX`; review both `monitoring/prometheus/rules/` and the
  staged Helm copies for exactly that.

## Migration Plan

1. Add the floor to both `monitoring/slo/*.slo.yaml`; run `monitoring/slo/generate.sh`;
   commit the regenerated `monitoring/prometheus/rules/*` and staged
   `deploy/helm/*/files/slo/*` (diff = floor only).
2. Update `tests/tenant-router.slo_test.yaml`; run `monitoring/slo/check.sh` → green.
3. Deploy is a rules-only change (no producer redeploy). **Rollback:** revert the source
   edit + regenerate; the previous rules return.
4. Validate: below 60 samples/window the SLI yields no series so the burn-rate alerts cannot
   fire (the 5m arm reproduces the ~0.2-rps anchor); above it, SLI values and objectives are
   unchanged; readiness alerts unaffected.
5. Mark N15 resolved in `docs/infra-findings.md` (correcting its "source not in repo" claim).
