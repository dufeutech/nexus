Routing keys on the request HOST. Only VERIFIED domains resolve (RFC C16); an
unknown/unverified host is rejected at the edge with 404 (RFC C18) — never
defaulted to a tenant.

Register routing config via the control plane (administrative boundary — NOT public):
  kubectl -n {{ .Release.Namespace }} port-forward svc/{{ include "routing-plane.fullname" . }}-control-plane 9400:9400 &
  curl -X POST localhost:9400/tenants -H content-type:application/json \
       -d '{"tenant_id":"acme","plan":"pro","target_pool":"application"}'
  curl -X POST localhost:9400/domains -H content-type:application/json \
       -d '{"domain":"app.acme.example.com","tenant_id":"acme","verified":true}'

Smoke test (from inside the cluster, once a verified domain is registered):
  curl -s -H 'Host: app.acme.example.com' http://{{ include "routing-plane.fullname" . }}-edge.{{ .Release.Namespace }}.svc/ | grep -i '^x-tenant'
  # expect x-tenant-id / x-tenant-plan / x-route-pool injected, routed to its pool

For the tenant-first composition WITH the identity plane (routing -> auth ->
enrichment on one edge), use the ../edge-platform umbrella chart instead.

External dependencies (operated OUTSIDE this chart): see deploy/README.md
  - Postgres (the authoritative routing store; HA is your operator's concern)
  - Redis (optional L2; the plane is correct without it)
  - The backend pools the routing decision selects among (pools.*)

UPGRADE NOTE (chart 0.2.0) — control-plane metrics moved port:
  * /metrics + /healthz are now on a dedicated ops port :9401 (off the broker-only
    admin :9400). The bundled ServiceMonitor already targets :9401, but any
    EXTERNAL Prometheus / scrape config pointed at :9400/metrics will silently go
    dark — retarget it to :9401.
  * This needs the rebuilt control-plane image (it now listens on :9401). The
    chart's default image tag tracks appVersion (0.2.0), so build & push the
    first-party images at that tag before upgrading, or the ops probe CrashLoops.
